# RD Spec / Plan 產出技能

## Overview

引導工程師逐步填寫標準 Spec/Plan 文件，完成後輸出 `.md` 檔案。

## 使用時機

- 開始開發新功能前，需要撰寫技術規格
- PM 提供 Requirement Brief 後，需要產出 Spec/Plan
- Sprint 規劃時，需要估算工時與定義驗收清單

## 執行流程

### Step 1：蒐集 Header 資訊

詢問或從對話中取得以下欄位：

| 欄位 | 說明 |
|------|------|
| 功能名稱 | 與 Requirement Brief 一致 |
| Brief 連結 | Requirement Brief 的連結或編號 |
| Owner (RD) | 負責人姓名 |
| 歸屬模組 | 例：InsightVex / NL2SQL 引擎 |
| Spec 日期 | YYYY-MM-DD |
| Sprint | 例：Sprint #12 |
| Review 狀態 | 草稿 / Review 中 / 已通過 |

### Step 2：逐節引導填寫

依序引導以下六個章節。若使用者提供 PM brief，先用 Step 0 快速草稿再逐節確認。

#### 一、需求理解確認

**填寫重點：**
- 用工程師自己的話重述 PM brief 的目標
- 說明：解決什麼問題、給誰用、做到什麼程度算完成
- 這是 Spec Review 時 PM 確認理解正確與否的依據

**範例格式：**
```
客戶的工單目前人工分派，我們需要做一個基於 LLM 的自動分派系統，讓工單根據內容
自動分配到最適合的客服人員，目標準確率 90% 以上，延遲 5 秒內。
```

#### 二、技術方案

**2.1 架構選型**
說明核心技術選擇及理由，包含與替代方案的 trade-off。

**2.2 核心流程**
用文字或流程圖描述主要處理步驟（進入 → 處理 → 輸出）。

**2.3 資料模型**
列出新增或變更的 DB schema：

| 欄位 | 型別 | 必填 | 說明 |
|------|------|------|------|
| (欄位名) | (型別) | Yes/No | (說明) |

**2.4 API 設計**
列出新增或變更的 API endpoint：

| Method | Endpoint | Request | Response |
|--------|----------|---------|----------|
| POST | /api/v1/... | { ... } | { ... } |

**2.5 與現有系統整合**
說明如何與現有模組整合，是否會影響其他功能。

#### 三、任務拆解與時間估算

**填寫原則：**
- 每個子任務應可在 4 小時（半天）內完成，超過則繼續拆分
- 估算包含 AI Coder 輔助後的時間
- 估算偏差目標 ≤ 20%

| 子任務 | 預估工時 | 實際工時 | 偏差 | 說明 |
|--------|----------|----------|------|------|
| (任務名) | Xh | (完成後填) | - | (說明) |
| **總計** | **Xh** | | | |

> 「實際工時」和「偏差」欄在 Sprint 結束後填寫。

#### 四、生產環境目標

**填寫原則：**
- 數字是績效承諾，應對齊 PM brief 的產品指標目標
- 若認為目標不可行，在 Spec Review 時提出並協商

| 指標 | 目前 Baseline | 本次交付目標 |
|------|--------------|-------------|
| **正確性** | | |
| 例：準確率 | (現況) | ≥ X% |
| 例：幻覺率 | N/A（新功能） | ≤ X% |
| **速度** | | |
| 例：延遲 P50 | (現況) | ≤ Xs |
| 例：延遲 P95 | (現況) | ≤ Xs |
| **效率** | | |
| 例：Token 成本/次 | N/A | ≤ US$X |
| 例：快取命中率 | N/A | ≥ X%（上線 2 週後） |

#### 五、風險與相依

| 風險 | 機率 | 影響 | 緩解方案 |
|------|------|------|----------|
| (風險描述) | 高/中/低 | (影響說明) | (緩解方式) |

**外部相依：** 列出依賴其他人員或外部資源的項目。

#### 六、驗收清單

**功能驗收**（對應 PM brief 驗收標準）：
- [ ] (驗收標準 1)
- [ ] (驗收標準 2)

**生產環境驗收**：
- [ ] 正確性指標達到第四節承諾的目標值
- [ ] 速度指標達到第四節承諾的目標值
- [ ] 效率指標達到第四節承諾的目標值
- [ ] 監控 Dashboard panel 已設定並能正常採集數據

**品質驗收**：
- [ ] Code Review 通過
- [ ] 測試覆蓋率 ≥ 70%
- [ ] 靜態分析零 critical issue

## 快速產出 Prompt（附錄）

若使用者提供完整 PM brief，可先用以下 prompt 快速產出草稿，再逐節確認：

```
你是一位全端工程師。請根據以下 Requirement Brief，產出一份 Spec/Plan，包含：
1. 需求理解確認（用自己的話重述）
2. 技術方案（架構 + 核心流程 + DB schema + API 設計）
3. 任務拆解與工時估算（每個子任務 ≤ 4 小時）
4. 生產環境目標（正確性 / 速度 / 效率，含具體數字）
5. 風險與緩解方案
6. 驗收清單

我們的技術栈：[... 填入技術栈 ...]

Requirement Brief：
[... 貼上 PM 的 brief ...]
```
